# Agent Flow â€” VS Code Extension

## What This Is

Agent Flow is a VS Code extension that visualizes Claude Code's multi-agent workflows as a real-time, interactive, force-directed graph. It reads Claude Code's JSONL transcript files from `~/.claude/projects/`, parses them into a node-edge graph (orchestrator â†’ agents â†’ tool calls), and renders them on an HTML5 Canvas inside a VS Code webview panel.

The target user is a developer running Claude Code who wants to see what their agents are doing: which sub-agents were spawned, what tool calls each one made, which files were touched, how much it cost, and the ability to replay a completed session step-by-step.

### sessionWatcher.ts (Live Mode)

Uses `setInterval`-based polling (NOT `fs.watch` â€” that's unreliable on macOS for detecting file appends). Every 400ms, it `fs.stat()`s the watched JSONL file. If size or mtime changed, it reads only the new bytes from the end of the file and emits a `'newEntries'` event.

The extension host listens for `'newEntries'`, then re-parses the entire file (debounced at 200ms) and sends the updated `ParsedSession` to the webview via `postMessage({ command: 'liveUpdate', data })`.

Also polls the subagents directory (`<session-id>/subagents/`) for new/changed `.jsonl` files.

Key: `constructor(claudeDir, pollIntervalMs=500)` â€” the interval is configurable.

### jsonlParser.ts (JSONL â†’ Graph)

Reads the entire `.jsonl` file synchronously. Each line is one JSON object.

**JSONL entry types it handles:**
- `type: "user"` â†’ creates a `user` node, edge to orchestrator
- `type: "assistant"` â†’ scans content blocks for `tool_use` and `text`
  - `tool_use` with `name: "Task"` â†’ creates an `agent` node (sub-agent spawn)
  - `tool_use` with any other name (Bash, Read, Write, Edit, Grep, etc.) â†’ creates a `tool` node
  - `text` matching `/launch.*agents/i` â†’ creates an `assistant` node
- `type: "agent"` or entries with `agentId` field â†’ creates `agent` nodes
- Token usage from `message.usage.{input_tokens, output_tokens}` â†’ accumulated for cost

**Subagent transcripts:** Also reads `<session-id>/subagents/agent-<id>.jsonl` files and attaches their tool calls as children of the corresponding agent node.

**File touch tracking:** Tools named Read, Write, Edit, MultiEdit, Grep, Glob are tracked in `touchedFiles[]` with the file path, action type, and which agent did it.

**Cost calculation:** Uses Sonnet 4.6 pricing: $3/M input tokens, $15/M output tokens.

**Exports:**
- `parseJsonlFile(filePath) â†’ ParsedSession`
- `discoverSessions(claudeDir) â†’ { label, filePath, date }[]`

**Key interfaces:**
```typescript
interface GraphNode {
  id: string;
  type: 'orchestrator' | 'user' | 'assistant' | 'agent' | 'tool';
  label: string;
  detail: string;
  timestamp: string;
  toolName?: string;
  parentId?: string;        // used to build tree structure
  // ... tokens, cost, agentId, agentType, status
}

interface ParsedSession { sessionId: string; nodes: GraphNode[]; edges: GraphEdge[]; totalTokens: number; totalCost: number; agentCount: number; startTime?: string; isLive: boolean; touchedFiles: FileTouch[]; }
```

### extension.ts (Extension Host + Webview)

The extension host code is ~80 lines at the top. The rest (~1070 lines) is three template literal strings: `CSS`, `HTML`, `JS` that form the webview content.

**Extension host responsibilities:**
- Registers two commands: `agentFlow.openPanel` and `agentFlow.selectSession`
- Creates a `WebviewPanel` with `enableScripts: true` and `retainContextWhenHidden: true`
- Handles messages from webview: `requestSessions`, `loadSession`, `startLive`, `stopLive`, `autoDetect`
- `startLiveMode(filePath)` creates a `SessionWatcher`, debounces at 200ms, re-parses and sends `liveUpdate` messages

**Webview JS â€” Major Systems:**

1. **Layout** (`initialLayout`): Weighted radial tree. Computes `subtreeWeight()` for each node (total descendants + 1). Children are allocated angular arc proportional to their subtree weight. Ring radius increases with depth. Agents placed first (from top), then other nodes, then users.

2. **Physics** (`applyPhysics`): Force-directed simulation running every frame.
   - All-pairs repulsion (Coulomb, scaled by node radius)
   - Edge springs (Hooke's law, target length varies by node types: orchestrator edges=200, agent edges=150, tool edges=160)
   - Center gravity (gentle pull toward graph centroid)
   - Damping 0.88/frame, velocity clamped to 8
   - Auto-settles after 60 frames of low energy, reactivates on node drag or data change

3. **Particles** (`spawnParticles`, `updateParticles`, `drawParticles`): Glowing dots travel along edges from source to target node. Each particle has position `t` (0â†’1), speed, alpha, size. Rendered with `shadowBlur` glow + trailing line segment.

4. **Replay Engine** (`buildReplayData`, `applyReplayCursor`, `advanceReplay`, `enterReplay`, `exitReplay`):
   - Parses all node timestamps into milliseconds, sorts them
   - `replay.cursorMs` tracks current position (ms from session start)
   - `applyReplayCursor()` builds a `visibleIds` Set and filters nodes/edges
   - `advanceReplay(dtMs)` advances cursor by `dt * speed` each frame when playing
   - Scrubber bar: click/drag to seek, dots positioned proportionally by timestamp
   - Speeds: [0.25, 0.5, 1, 2, 4, 8]x
   - Entrance animation: nodes scale up from 0.3â†’1.0 over 600ms when they appear

5. **Rendering** (`render`, `drawNode`, `drawEdge`, `drawHexagon`, `drawStarField`):
   - Canvas 2D rendering at 60fps via `requestAnimationFrame`
   - Orchestrator/agent = hexagons with dashed stroke and glow
   - Tool/user = circles
   - Star field background with parallax and twinkle
   - Pan (drag canvas), zoom (scroll wheel), drag nodes (click-drag on node, pins it)

6. **Side Panel Views:**
   - **Graph** (default): Canvas visualization, no side panel
   - **Files** (`buildFilesView`): Groups `touchedFiles` by directory, shows action badges (READ/WRITE/EDIT/GREP/GLOB), color-coded, with agent attribution
   - **Chat** (`buildChatView`): Scrollable transcript of all events with role labels and monospace formatting
   - **$Cost** (`buildCostView`): Total cost summary, bar chart of tool usage frequency, per-agent breakdown with tool counts
   - **Timeline** (`buildTimelineView`): Chronological event list with relative timestamps, colored dots, agent section headers

**Node colors:**

## Claude Code JSONL Data Source

Claude Code stores sessions at `~/.claude/projects/`. Directory names are the project path with `/` replaced by `-` (e.g., `-Users-tagtaste-Documents-myproject`). Each session is a UUID-named `.jsonl` file.

**JSONL line format:**
```jsonl

{"type":"assistant","timestamp":"2025-03-24T10:30:15Z","message":{"role":"assistant","content":[{"type":"text","text":"Let me investigate..."},{"type":"tool_use","name":"Bash","id":"tool_1","input":{"command":"grep -r 'auth' src/"}}],"usage":{"input_tokens":1500,"output_tokens":200}}}
{"type":"assistant","message":{"content":[{"type":"tool_use","name":"Task","id":"task_1","input":{"description":"Explore autoscaling config","prompt":"..."}}]}}
{"type":"summary","summary":"Fixed authentication middleware..."}
```

Subagent transcripts are at: `<session-id>/subagents/agent-<id>.jsonl`

## Known Issues and Limitations

1. **Cost view shows tool call counts, not actual per-agent token cost** â€” the JSONL doesn't always attribute tokens to specific agents cleanly. The parser accumulates tokens globally. Per-agent token tracking would require correlating tool_use IDs with subsequent tool_result messages, which isn't implemented yet.

2. **Layout can overlap** with very large sessions (100+ tool nodes) â€” the force-directed physics handles it eventually but initial positions may cluster. Could benefit from a Barnes-Hut optimization for O(n log n) repulsion instead of O(nÂ˛).

3. **All webview code is inline** in `extension.ts` as template literal strings. This works but makes editing the CSS/HTML/JS awkward (no syntax highlighting inside template literals). Could be extracted to separate files in `media/` and loaded at runtime.

4. **No persistence** â€” camera position, zoom level, and panel state are lost when the webview is hidden and restored. Could use `vscode.getState()`/`setState()`.

5. **Replay mode exits on live updates** â€” by design, but could be improved to let the user stay in replay while new data arrives.
